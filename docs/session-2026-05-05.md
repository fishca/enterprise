# Session 2026-05-05 — paging architecture handoff

Long, exploratory session. The arc started from "wire frontend through
the existing paged buffer", went through real Phase 4 cuts on three
list-class concrete models, then through a tree-paging detour on
FolderRef, and converged on a universal `Get*Fetch` contract that
backs desktop, the web frontend, and any future headless caller off
the same backend API.

Canonical design — `docs/paging-design.md`, especially §8 «Evolved
architecture». This file is the session-level handoff with the
specifics of what changed in code and where the next session picks up.

## What landed

### 1. Tape paging on table-class concrete models ✅

`ibValueListDataObjectRef`, `ibValueListDataObjectEnumRef`,
`ibValueListRegisterObject` all expose typed `Fetch(req)` and override
`IsPaged() = true` / `NotifyViewportChanged` / `ResetForFilterOrSort`
to forward into a per-instance `m_buffer`
(`ibTableViewBuffer<TKey, TRow>` in `tableInfo.h`).

* Ref — full cursor pagination (anchor + sort tuple). Forward / Backward
  fetch, viewport+1 cap, eviction in tape style.
* Enum / Register — single-batch (CASE/WHEN order and composite-key
  cursoring left for later).

### 2. `ibListSqlBuilder` ✅

`enterprise/src/engine/backend/metaCollection/partial/list/listSqlBuilder.{h,cpp}`
— `BuildSelectHead` / `AppendLimit` / `BuildFilterWhere` /
`BuildOrderBy` / `BuildAnchorPredicate` / `BindFilterParams` /
`BindAnchorSortParams`. Static helpers shared between the new paged
`Fetch` and the legacy `RefreshModel` so we don't keep copying ~60
LOC of WHERE / ORDER-BY building per concrete model.

Takes `const ibValueMetaObjectRecordDataRef*` (where the predefined
`IsDataReference` lives). Register branches inline its own filter +
order — `ibValueMetaObjectRegisterData` is a separate hierarchy with
no `IsDataReference`.

### 3. Frontend dispatch via `IsPaged()` ✅

`ibValueModel::CallRefreshModel` and `CallRefreshItemModel` route
based on `IsPaged()`:

* paged → convert frontend's `(topItem, count, scroll)` into
  `ibViewport` and call `NotifyViewportChanged`.
* non-paged → existing `RefreshModel` / `RefreshItemModel` path.

`tableBoxEvent.cpp` not touched. The switchover is invisible to the
frontend control wrapper.

### 4. `ibPagingLog` diagnostics ✅

`BACKEND_API void ibPagingLog(const char* fmt, ...)` —
`wxLogDebug` + appends a timestamped line to `paging.log` in the
working directory. Wired across buffer entry points
(`NotifyViewportChanged`, `ResetForFilterOrSort`, `DispatchReset` /
`Forward` / `Backward`, `Apply`) and all Fetch impls (entry args +
SQL for Ref + post-exec `rows / hasMore`). Open the file alongside a
running enterprise / launcher process to trace paging behaviour
without attaching a debugger.

### 5. wxDVC sortOrder assert fix — `HasDefaultCompare = !IsPaged()` ✅

wxDVC's `ibDataViewTreeNode::InsertChild` (datavgen.cpp:1469) asserts
`m_branchData->sortOrder == sortOrder` ("Logic error in wxDVC sorting
code") when our `Insert(0)` (backward fetch) lands after `Append`
(forward fetch) under a model that claims `HasDefaultCompare = true`.
For paged tables we own row order ourselves; flipping the override to
`!IsPaged()` makes wxDVC stop tracking branch sort, the assert ducks.
Header click sort still works — it routes through the next item.

### 6. Direct sort integration via fork ✅

New virtual `ibDataViewModel::OnSortColumnChanged(col, ascending)`.
Fork's column-header click handler (`datavgen.cpp` ~358) calls it
directly instead of `model->Resort()`. Default impl forwards to
`Resort()` (legacy non-paged behaviour); paged concrete models override
to update `m_sortOrder` and trigger `ResetForFilterOrSort`. Sort works
end-to-end in paged mode without the wx event ping-pong.

### 7. `RefreshFromAction` ✅

Single helper on `ibValueModel`: paged → `ResetForFilterOrSort`,
non-paged → `CallRefreshModel`. Used from the four `enRefresh`
handlers in `objectList.cpp` and the three filter cases in
`tableInfo.cpp::ExecuteAction`. Refresh button now reloads paged
catalogs correctly, filter actions propagate.

### 8. FolderRef tree (legacy load-all path improved) ✅

* SQL `ORDER BY isFolder_B DESC, parent_RRRef ASC, uuid ASC` so
  folders bubble to the top in any view mode. Sort logged in
  `paging.log`.
* Client-side build through parent-guid match (existing) plus
  hierarchical filter so `m_topParentGuid` scopes the visible window.
* `GetAncestorChain(fromGuid)` — public helper, walks parent chain
  via `WHERE uuid = ?` until empty parent. Cached on
  `m_topParentGuid` (`m_chainCachedFor` / `m_chainCache` mutable
  members) — repeat hits on the same drill no longer re-walk DB.
* `SetParentTopItem(item)` updates `m_topParentGuid`, dedupes on
  same-as-current, and triggers `RefreshFromAction()` so drill-down
  / drill-up reload the tree under the new top.

### 9. Typed `Get*Fetch` API on FolderRef ✅

`GetFirstFetch(args) / GetNextFetch(args) / GetPrevFetch(args)` with
`ibTreeFetchArgs { m_parent, m_currentRow, m_viewportAnchor,
m_count }` and `ibTreeFetchResponse { m_rows, m_hasMore }`.
wxTreeCtrl iteration cookie pattern — `m_viewportAnchor` is the
cursor. SQL: `WHERE parent = ? AND uuid > ?` (Next) / `< ? DESC` then
reverse in memory (Prev). `GetFirstFetch` is a thin wrapper over
`GetNextFetch` with cleared anchor.

### 10. Refcount-aware `ibDataViewItem` ✅

`tableView.h` — `ibDataViewItem` now stores `wxRefCounter*` and does
auto IncRef on copy / move-ctor / value-ctor, DecRef on dtor. Existing
`ibDataViewItem(void*)` ctor preserved (cast inside) for ABI back-compat
because every OES row class (`ibValueTreeNode`, `ibValueTableRow` and
their subclasses) inherits from `wxRefCounter` already. Provides
`IsOk` / `GetID` / `operator bool` / `operator!` / `==` / `!=`.

Solves selection-holds-evicted-row, frozen breadcrumb survival across
model rebuilds, dangling pointers after `Clear()` / batch replacement.

Single legacy callsite found in `predefinedEditor.cpp` — accessed
private `m_pItem` member; rewritten to `item.GetID()`.

### 11. Universal `Get*Fetch` virtuals ✅

`ibDataViewModel` and mirror on `ibValueModel` get
`GetFirstFetch / GetNextFetch / GetPrevFetch` virtuals. Default
fallbacks: `GetFirstFetch` → `GetChildren` (legacy non-paged models
keep working), `Get(Next|Prev)Fetch` → 0 (no paging support). Const
signatures so callers holding `const ibDataViewModel*` (the fork's
`BuildXxxHelper`) can use them.

`ibValueModel::ibDataViewModelProviderImpl` forwards each to the
owner ibValueModel.

### Scroll polish — punch list

* **Buffer-edge fetches racy with eviction.** When `kEvictTrigger`
  fires it re-bases `m_pagedFwdAnchor / m_pagedBwdAnchor` to the new
  edges of the loaded children list, but the tape balance on the next
  tick may already have decided to refill from the deficit side using
  the pre-eviction state — leads to one wasted fetch round trip.
* **`m_pagedHiddenTail` re-base after `PagedFetchForward`** un-hides
  every previously-tail-hidden row instead of sliding the hidden
  region to track the new end (`datavgen.cpp:3976-3985`). Visible
  count briefly balloons before the next OnScroll re-hides them.
* **THUMB drag** triggers a long sequence of `LINEUP / LINEDOWN`
  ticks; each calls a fetch when the hidden buffer empties. A
  fast-drag past the loaded window blanks the tail until N fetches
  catch up. Either coalesce ticks (idle-deferred fetch) or detect
  THUMB and do a Reset + bootstrap around the new top.
* **Initial bootstrap + restore selection** — `PagedBootstrap` picks
  selection over top, but if both are set and far apart the restored
  window is centred on selection, not on the user's prior scroll
  position. Acceptable today; revisit when keyboard navigation lands.
* **Mouse-wheel** doesn't take the scroll path on every wxDVC
  variant; verify `EVT_SCROLLWIN` fires under wheel + that the
  derived dir (LINEUP/LINEDOWN) is set correctly.
* **`m_pagedInFlight` is a counter, not a per-direction guard** —
  two near-simultaneous OnScroll events in opposite directions only
  block each other, never serialise into independent fetches.
  Probably fine while DoFetch is sync; revisit when async lands.

## Open audits

* **Refcount + shared_ptr mixing.**
  `ibPredefinedValueObject` is `shared_ptr`-managed and inherits
  `wxRefCounter`. Adding `ibDataViewItem` IncRef on it does not extend
  the `shared_ptr`'s lifetime; `shared_ptr` can delete while the item
  still holds, then the item's DecRef hits a freed object. Either
  drop refcount handling for shared_ptr-managed types or unify
  ownership before the refcount-aware item is exercised on those
  flows.
* **In-flight fetch cancellation.** Async Fetch needs a generation
  token on `ibFetchArgs` (or `ibSession::RequestCancel`) so a result
  whose viewport context already moved on gets discarded instead of
  applied.

## Decisions to remember

* **Three Fetch methods (First / Next / Prev), not one with a
  direction flag.** Clearer at call sites; mirrors the wxTreeCtrl
  iteration cookie pattern (`GetFirstChild + cookie + GetNextChild`).
* **Model is stateless w.r.t. fetched rows.** Each `Get*Fetch` call is
  independent. The model creates rows and hands them off; the caller
  (GUI, web, headless script) keeps them alive via
  `ibDataViewItem`'s refcount. Whatever lives in
  `ibValueModelTreeBase::m_root` etc. is legacy-load-all fallback,
  not the paged path.
* **`m_currentRow` ≠ `m_viewportAnchor`.** User selection survives
  scroll; viewport edge moves with scroll. The args carry both —
  selection for post-fetch focus retention, viewport for the SQL
  cursor.
* **wxDataView fork is ours.** Where the original wx interface forces
  contortions (sort assert, branch sortOrder tracking, helpers that
  call `GetChildren` instead of paged Fetch), modify the fork
  directly rather than work around it from the model side.
* **Per-scroll round-trip is acceptable on local FB.** Async via
  worker pool is the answer for remote DB; not blocking on it for
  Phase 4 ship.
