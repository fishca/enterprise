# Session 2026-05-06 — paging architecture, control-driven prefetch

Continuation of the 2026-05-05 universal `Get*Fetch` arc. The 05-05 cut
landed the contract on the model side; 05-06 inverted the control's
role from "ask the model for Refresh" to "drive its own prefetch deque
against a stateless fetch service" and closed every Scroll-polish item
from the previous handoff.

Canonical design — `docs/paging-design.md` §8 (status table refreshed
in §8.8). This file is the session-level handoff for the items that
landed on 05-06 specifically.

Working copy at end of session is uncommitted (per project convention
— user reviews diff before commit). Build clean, smoke OK against
`fb_test251` catalogs / documents.

## What landed

### 1. `ibDataViewObject` base for rows and nodes ✅

New base in `tableView.h` between `wxRefCounter` and the row/node
hierarchy. Three virtuals replace the previous "ask the model" indirection:

```cpp
class BACKEND_API ibDataViewObject : public wxRefCounter {
public:
    virtual bool IsEqualTo(const ibDataViewObject&) const;   // pointer match by default
    virtual bool IsContainer() const;                         // false by default
    virtual ibDataViewItem GetParentItem() const;             // empty by default
};
```

`ibDataViewItem` is now a value-type wrapper around
`ibDataViewObject*` with auto IncRef/DecRef on copy/move/dtor and
`operator==` / `operator!=` / `operator bool` / `IsContainer` /
`GetParentItem` delegation. Legacy `ibDataViewItem(void*)` ctor stays
for the one non-`ibDataViewObject` holder (`ibDataViewTreeStoreNode`,
delete-owned) and for ABI back-compat — internal cast to
`ibDataViewObject*`.

`ibValueTableRow` now derives from `ibDataViewObject` (was
`wxRefCounter`); overrides `IsEqualTo` to compare `m_nodeValues` so
freshly-fetched rows behind new pointers match the previously-selected
one.

`ibPredefinedValueObject` stays `wxRefCounter` (not part of the
data-view hierarchy — predefined values aren't model rows; nothing
passes them through `ibDataViewItem`).  The 05-05 mixing concern
turned out to be a false alarm: confirmed no callsite hands a
`ibPredefinedValueObject*` to `ibDataViewItem`, so the
`shared_ptr` lifetime management on it never fights with item
refcount.

### 2. Concrete `Get*Fetch` overrides on every paged model ✅

Each of EnumRef / Ref / RegisterObject / FolderRef now exposes the
universal trio (`GetFirstFetch / GetNextFetch / GetPrevFetch`) as
`override` and keeps a private typed `Fetch(const ibFetchRequest<TKey>&)
const` as the single SQL point. Legacy `RefreshModel` /
`RefreshItemModel` virtuals removed from these classes. `IsPaged()`
is `true` on all four — even single-batch ones (Enum, Register) —
so the control drives its deque uniformly.

### 3. Lazy tree expansion ✅

`ibValueTreeListNode::LoadState { NotLoaded, Loading, Loaded }` lives
on the node, mutable so a const `GetChildren()` can drive the
first-expand fetch without const_cast'ing the whole node. Folder
flag (`m_container`) wins over the base "has children loaded" check
so a paged FolderRef row knows it CAN contain children even before
they're populated.

### 4. Single refresh entry point — `RefreshFromAction()` ✅

```cpp
void ibValueModel::RefreshFromAction() {
    if (m_modelProvider) {
        m_modelProvider->BeforeReset();
        m_modelProvider->AfterReset();
    }
}
```

Replaces `CallRefreshModel(top, count)` /
`CallRefreshItemModel(top, current, count, scroll)` /
`m_refreshModel` flag. `RefreshModel` / `RefreshItemModel` virtuals
also gone. The control's `Cleared()` walks the notifier chain, the
control then re-fetches via `GetFirstFetch`. One source of truth for
"data changed, redraw".

`OnSortColumnChanged` updates `m_sortOrder` for paged models, then
calls the same notifier path. Non-paged models route to legacy
`Resort()`.

`tableBoxEvent.cpp::OnIdle` shrunk to layout column-position
calculation only; the previous fetch + selection-restore lattice is
gone (control owns it now). `HandleOnScroll` removed entirely.

### 5. Control-driven prefetch in `ibDataViewCtrl` ✅

Most of the 950-LOC growth in `datavgen.cpp` is here. The control owns
the buffer; the model is a stateless `Get*Fetch` service.

* `OnScrollEvent(wxScrollWinEvent&)` — sole scroll-driven dispatcher.
  Replaces `ibValueModelTableBox::HandleOnScroll`.
* `OnIdleEvent(wxIdleEvent&)` — handles `m_pagedNeedsBootstrap`.
* `PagedBootstrap()` — first `GetFirstFetch` once layout settles.
* `PagedFetchForward(int batch)` / `PagedFetchBackward(int batch)`
  — extend the loaded window when scroll approaches an edge.
* `OnPagedFetchForwardResult / OnPagedFetchBackwardResult` —
  UI-thread result processors invoked via `CallAfter` after the
  async DB roundtrip completes on the worker.
* `PagedRefresh(preferSelection)` — wipe loaded window and queue a
  fresh `GetFirstFetch` on next idle. Universal "model said something
  changed" handler (ItemAdded / ItemDeleted / DoItemChanged in
  paged mode).

State on `ibDataViewCtrl`:

```cpp
bool           m_pagedNeedsBootstrap = false;
bool           m_pagedHasMoreFwd     = false;
bool           m_pagedHasMoreBwd     = false;
int            m_pagedFetchingFwd    = 0;          // per-direction guard
int            m_pagedFetchingBwd    = 0;
uint64_t       m_pagedFetchGen       = 0;          // generation token
ibDataViewItem m_pagedFwdAnchor;                   // cursor, not derived
ibDataViewItem m_pagedBwdAnchor;                   //   from m_root.GetChildNodes()
ibDataViewItemArray m_topParentChain;              // refcount-pinned drill chain
```

### 6. Generation-token cancellation ✅

`m_pagedFetchGen` is bumped on every full reset (`PagedRefresh` /
`Cleared` / `AssociateModel`). Each fetch lambda captures the token
at submit time; on completion it checks against the live token and
discards the result if the user has moved on (sort change, filter
change, model swap). Closes 05-05's «In-flight fetch cancellation»
open audit.

### 7. Per-direction in-flight counters ✅

Replaces 05-05's single `m_pagedInFlight` counter that "blocks each
other instead of serialising". `m_pagedFetchingFwd` and
`m_pagedFetchingBwd` are independent — a scroll burst that crosses
both edges can dispatch to each side without serialising.

### 8. Soft-eviction via `wxDataViewTreeNode::SetHidden` ✅

```cpp
bool IsHidden() const { return m_hidden; }
void SetHidden(bool h) {
    if (m_hidden == h) return;
    m_hidden = h;
    if (m_parent) m_parent->ChangeSubTreeCount(h ? -1 : +1);
}
```

Hidden nodes stay in `m_children` but are skipped by walkers and
excluded from `subTreeCount`. Backward scroll re-shows previously
fetched rows from the hidden head instead of going to DB again.
Closes 05-05's «`m_pagedHiddenTail` re-base un-hides everything»
punch-list item.

### 9. 3-state lying scrollbar ✅

THUMB drag was the worst offender from 05-05's punch list — fast drag
chained many `LINEUP/LINEDOWN` ticks, each kicking a fetch.

Fix is structural — the scrollbar lies. Thumb has 3 fixed positions
({Top, Middle, Bottom}) derived from `m_pagedHasMoreFwd /
m_pagedHasMoreBwd`. Fork overrides:

* `IsPagedScrollbarMode()` / `DerivePagedThumb()` —
  enRoute is "paged + viewport-aware".
* `UpdatePagedScrollbar()` — applied at every `wxScrollHelper`
  update path so its real-virtual-size values cannot leak through.
* `AdjustScrollbars()` / `SetScrollPos()` /
  `SetScrollbar(orient, pos, thumbSize, range, refresh)` — virtual
  overrides that re-route to the lying scrollbar.

Side benefit: Scrollbar UX matches the data model — "more above /
more below" is what the user sees, not a fake position derived from
loaded-window count.

### 10. wxDVC sort suppression for paged ✅

```cpp
SortOrder GetSortOrder() const {
    if (GetModel() != nullptr && GetModel()->IsPagedModel())
        return SortOrder();   // empty — no sort
    // ... legacy column-derived sort
}
```

For paged models the SQL `ORDER BY` drives the rows we receive;
telling wxDVC to also sort would make `ItemAdded / InsertChild`
route through binary-search insertion and scatter freshly fetched
rows across the tree. Empty `SortOrder()` makes wxDVC append in
fetch order. Closes the «sort assert» class of bugs at the source
rather than working around it from the model.

### 11. Hierarchical drill chain pin ✅

`m_topParentChain` (refcount-pinned `ibDataViewItem` array) survives
`Cleared() / DestroyTree` that wipes the previous `m_root`. Front
element is the current top (the folder the user is currently inside);
subsequent elements are its ancestors up to the dataset root.
`GetTopParentItem()` reads front of chain. Replaces a single anchor
that was wiped on re-fetch (drill-up after drill-down lost the
breadcrumb context).

### 12. Async via `ibSession::Submit` end-to-end ✅

```cpp
// ibDataViewModel virtual
virtual void SubmitFetchAsync(std::function<void()> work) {
    if (work) work();   // default — inline (legacy non-paged)
}

// ibValueModel override (in tableInfo.h)
std::future<void> SubmitFetchAsync(std::function<void()> work);
//   → forwards to ibSession::Submit; inline fallback when no session
```

The control submits `PagedFetchForward / Backward` work via the
model's `SubmitFetchAsync`; the worker thread runs the typed `Fetch`,
then the lambda hops back to the UI thread via `CallAfter` and feeds
`OnPagedFetchForwardResult / OnPagedFetchBackwardResult`. The model
exposes both signatures: `void` (used by the control via base
virtual) and `std::future<void>` (used by callers that want to
join). Inline fallback when no session is bound matches
`ibSession::Submit`'s no-pool contract.

### 13. `AdoptRowsToItems` template helper

Already landed 2026-05-05 but worth re-noting: every typed →
universal `Get*Fetch` bridge uses it to transfer the initial allocation
refcount cleanly:

```cpp
template <class TRow>
static void AdoptRowsToItems(std::vector<TRow*>& rows,
                             ibDataViewItemArray& out) {
    for (auto* r : rows) {
        out.Add(ibDataViewItem(r));   // IncRef → 2
        r->DecRef();                  // back to 1, owned by item
    }
    rows.clear();
}
```

## Open audits

* **Selection survival under deep tree restructure.** `IsEqualTo` on
  `ibValueTableRow` compares `m_nodeValues` — works for catalog rows
  where the full row is in the values map. For tree nodes
  (`ibValueTreeListNode`) the comparison is pointer-only via the base
  default. If a re-fetch hands a fresh node pointer for the
  previously-selected folder, selection won't transfer. Either
  override `IsEqualTo` on tree-node classes (compare `m_objGuid`)
  or accept the limitation.
* ~~**`m_topParentChain` invalidation.**~~ Closed 2026-05-07 — code
  review confirmed the recovery path works without an explicit hook.
  Scenario: chain `[deep, mid, root]`, another user deletes `mid`.
  Drill-up `RemoveAt(0)` → chain `[mid, root]`, refresh fetches under
  the deleted `mid` → `GetFirstFetch` returns 0 rows → `PagedBootstrap`
  n==0 branch sets `hasFwd=hasBwd=false` cleanly (no phantom
  scrollbar). Next drill-up → `[root]` → normal fetch resumes. UX is
  one empty step then recovery; explicit per-entry validation would
  cost a round-trip per chain entry on every refresh and isn't worth
  it for the rare case.
* ~~**Generation token vs reentrant `Submit`.**~~ Closed 2026-05-07 —
  code review confirmed the chain is robust on both inline-fallback
  and worker-pool paths. All `m_pagedFetchGen` reads/writes happen on
  the UI thread: gen capture in `DispatchPagedFetch` (UI), gen bump in
  `PagedRefresh` / `Cleared` / `AssociateModel` (UI), gen check inside
  the `CallAfter`-marshalled lambda (UI). `CallAfter` posts to the
  next UI idle event so the check is sequential w.r.t. any
  intervening bump. Worker thread never touches gen. The «race
  between submit and bump completes» concern was theoretical; the
  multi-session load test will cover it observationally without a
  code change.

## Decisions to remember

* **Control owns the deque, model is stateless.** The 05-05 design
  said it; 05-06 enforces it. Backend has no notion of viewport,
  scroll position, or in-flight fetches. Every call into the model
  is self-contained given args. Web frontend will plug into the
  same surface — `/fetch` endpoint just marshals args and relays the
  rows.
* **Lying scrollbar over real scrollbar.** Trying to make
  `wxScrollHelper` reflect real cursor-paginated state (where total
  size is unknown until exhausted) leads to UX bugs (THUMB drag
  over phantom rows). 3-state lying scrollbar matches the data
  model: "is there more above / more below". Discoverable,
  controllable, doesn't lie *more* than necessary.
* **Sort suppression in wxDVC, not in the model.** Earlier instinct
  was to override `HasDefaultCompare` on the model. Real fix is at
  the fork: `GetSortOrder()` returns empty for paged models so the
  fork's `InsertChild` appends in fetch order. One-line fix that
  closes a class of bugs that would otherwise need per-row
  workarounds.
* **`ibDataViewObject` virtuals on the row, not on the model.** The
  earlier `wxDataViewModel`-style API put `IsContainer(item)`,
  `GetParent(item)` on the model — sensible when items are opaque.
  With refcount-aware items the row IS reachable, so virtuals live
  on the row. Smaller model surface, fewer `dynamic_cast` chains.
