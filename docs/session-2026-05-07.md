## Findings

### 1. Drill leaves virtual size stale → scroll one-viewport range

`ibDataViewCtrl::SetTopParent` (`datavgen.cpp:6403`) drives drill UP
and drill INTO through `Cleared() → BuildTree() → BuildHierarchical
Helper`. `Cleared()` ends with `UpdateDisplay()` which only flips
`m_dirty = true`; `RecalculateDisplay()` (where
`m_tableAreaWin->SetVirtualSize(width, GetLineStart(GetRowCount()))`
runs) is not invoked synchronously. wxScrollHelper kept the
pre-drill virtual size, computed scroll range = one viewport,
thumb stuck at top.

Fix: call `RecalculateDisplay()` right after `Cleared()` (and after
the no-op `PagedBootstrap` branch for legacy non-paged path),
inside the `Freeze()/Thaw()` pair so there's no flicker. Both drill
branches updated symmetrically.

### 2. FB stores parent ref as BINARY(20), not CHAR(16)

The session's first hypothesis ("FolderRef returns too few rows
under cursor because SQL parent filter is disabled") was correct but
underspecified. `objectListQuery.cpp:692` had a comment claiming FB
binary CHAR(16) "can't be filtered or bound reliably at SQL level"
and dropped the parent filter, doing matching in C++ over fetched
bytes. Three things were wrong with that:

* The column is **not** `CHAR(16)`. Reference attributes are saved
  via `metaAttributeObjectQuery.cpp:294` as
  `BINARY(reference_size_t)` where `reference_size_t = sizeof(ib
  Reference) = 20` (`[ibGuidImpl 16][ibMetaID 4]` on x86, packed).
  Bytes 16–19 hold the catalog's metaID in little-endian.
* `SetParamBlob` for FB **does** write into `sqldata` for
  `SQL_TEXT` columns (raw memcpy of N bytes — see
  `firebirdParameter.cpp:177-196`). Bind path was fine all along.
* For top-level rows whose logical parent is empty, save sets
  `_RTRef = 0` and `_RRRef = SetParamNull` — but FB rounds-trips
  the BINARY column as zero-padded with the m_id tail still
  filled in (because `AllocateParameterSpace` zero-inits sqldata
  but FB also re-uses prior parameter state on some bind paths).
  Stored value for top-level is
  `00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 <metaID LE 4>`.
  Not all-zeros. Not NULL. Not empty.

Diagnosed by adding probes that dumped the first 10 rows' `_RRRef`
through `GetResultBlob` (full 20 bytes, vs `GetResultString` which
truncates at first null) and tested literal `WHERE x = X'<hex>'` —
literal matched, param-bind didn't, until the bound blob's tail
matched the stored m_id.

Fix in `objectListQuery.cpp` for both `FolderRef::GetNextFetch /
GetPrevFetch`:

* Build `ibReference parentRefBlob{ m_metaObject->GetMetaID(),
  ibGuidImpl{} }` — m_id ALWAYS the catalog's metaID; m_guid all
  zeros for top-level, real GUID (MS-canonical, swapped from the
  big-endian `ibGuid::bytes()`) for drill.
* SQL: `SELECT FIRST N * FROM <table> WHERE <refDataField> = ?
  <cursor> ORDER BY ... LIMIT N`. Bind via `SetParamBlob(&blob,
  sizeof(ibReference))`.
* Drop the C++ byte matcher entirely. Drop the synthetic +1
  fetch-N math because the cursor now does its job — though
  `count + 1` probe row stays for `hasMore` detection.
* Add `BuildSelectHead` + `AppendLimit` calls that were missing
  for FolderRef altogether (FB embedded was returning every
  matching row regardless of `args.m_count` and clipping in
  C++ on `>= args.m_count`; PG/SQLite/MySQL/ODBC would have
  returned the entire table).

### 3. SQL_VARYING blob bind — bonus fix

While diagnosing #2 I hypothesised the FB driver might describe the
unnamed `?` parameter as `SQL_VARYING` rather than `SQL_TEXT`, and
`firebirdParameter.cpp::SetParamBlob` ctor only had branches for
`SQL_BLOB` and `SQL_TEXT` — VARYING was a silent no-op (sqldata
stayed zero-init, parameter looked like all-zeros to FB).

Telemetry showed `sqltype=452 = SQL_TEXT` for our case, so the
VARYING branch isn't on the hot path here. But the no-op was a
real latent bug for any future caller that binds blob-style data
to a varchar/varbinary column. Added the branch:

```cpp
else if (nType == SQL_VARYING) {
    const long cap = (long)m_pParameter->sqllen;
    const long n   = (nDataLength > cap) ? cap : nDataLength;
    ISC_USHORT len = (ISC_USHORT)n;
    memcpy(m_pParameter->sqldata, &len, sizeof(ISC_USHORT));
    memcpy(m_pParameter->sqldata + sizeof(ISC_USHORT), pData, n);
}
```

Also tightened the existing `SQL_TEXT` branch to clamp `n` against
`pVar->sqllen` (was unchecked memcpy — would scribble past the
allocation if a caller bound more bytes than the column declared).

### Late add — Hierarchical drill thumb / topRow / scroll position fixes

Series of related issues uncovered when testing scroll inside a
drilled subfolder.  Six layered bugs:

1. **PagedFetchForward / Backward inserted under `m_root`** instead
   of under the deepest crumb in the breadcrumb chain.  Same with the
   trim trigger which compared `m_root->GetChildNodes().size()`
   against `target = viewport + 2*kBufferSlack` — m_root has only
   the topmost crumb as a child in drill mode, so `over` was always
   negative and trim never fired.  Fix introduces `GetPagedInsertParent()`
   helper (walks `m_root → children[0]` × chain depth) and routes
   both result handlers through it; selection / currentRow /
   row-height-cache offsets are translated through `crumbCount` so
   flat-tree indices line up.
2. **`GetFirstVisibleRow()` returned data-area row index** in drill
   mode — `m_tableAreaWin->SetVirtualSize` subtracts the frozen-row
   offset from the height, so the target's logical y=0 corresponds
   to the first NON-frozen row.  `GetLineAt(y)` expects y in
   full-tree coords (row 0 = first row in m_root, frozen or not).
   Fix: shift y by `GetDataViewWindowOffset(m_tableAreaWin).y`
   before mapping y → row.  Mirrors what `DrawTableContent` does
   explicitly via the `gridOffset` argument to
   `CalcDataViewWindowUnscrolledPosition`.
3. **`GetRowByItem(GetTopItem())` returned -1** in drill mode.
   Fetched data rows have no model-side parent (paged Get*Fetch is
   stateless), so `ItemToRowJob`'s parent-chain walker fails to find
   them in the tree under their crumb.  Switched
   `OnScrollEvent`'s `topAdj` and `DerivePagedThumb`'s `topRow` to
   `GetFirstVisibleRow()` directly — bypasses the round-trip
   through item identity.
4. **dataBehind treated crumbs as data** — `topRow > 0` was true
   even at scroll-top in drill mode (topRow = crumbCount = 1).
   Crumbs are FROZEN at top and don't count as "data behind
   viewport".  Use `topRow > crumbCount`.
5. **Degenerate folder showed lying scrollbar** — `renderedRows >
   viewport` in `SetScrollbar` was true for a 14-row folder under
   one crumb (15 > 14) so a fake scrollbar appeared while data
   actually fit in the viewport.  Fix: subtract crumbCount before
   comparing.  Also added a corresponding clamp at the top of
   `DerivePagedThumb` to return Top when the dataset fits.
6. **Thumb stayed Bottom after scroll-up** — formula
   `topRow + viewport <= bufferSize` flickered between true / false
   in the overshoot region of m_tableAreaWin's virtual layout,
   forcing the user to scroll up two or three notches before the
   thumb left Bottom.  Replaced row-index math with direct
   scroll-position comparison: query `m_tableAreaWin->GetVirtualSize().y -
   GetClientSize().y` for max scroll, compare against
   `CalcUnscrolledPosition(0, 0, …)` result.  At scroll-bottom
   (`sy >= maxScrollY`) Bottom, at scroll-top (`sy <= 0`) Top, in
   between Middle.  Single scroll-up from Bottom now lands on
   Middle.

Flat (List / Tree) mode is a `crumbCount=0` degenerate of the same
formulas — identical behaviour.  Validated empirically via paging.log
diagnostics on a 114-row Document1004 list: thumb leaves Bottom on
the first scroll-up (`sy=2266 < max=2275`) and re-enters on
overshoot (`sy=2288 ≥ max=2275`).

### Tree+selection cursor — landed 2026-05-07 (late)

`SetViewMode` (List / Hier → Tree) with a deep selection produced
SQL `WHERE (fld_isFolder, …) <= (0, …)` that excluded all folders
from the top-level batch, so the tree-auto-expand walker missed at
depth 0 and selection lost expansion. Fix in `PagedBootstrap`
(`datavgen.cpp:4366-4368`): in Tree mode + restoreFromSelection use
empty cursor instead of selection cursor — top-level fetch starts
from the beginning (folders bubble first per `isFolder DESC`),
`BuildAncestorBreadcrumb`'s chain head lands in the buffer, walker
drills down to focus.

Path A — landed later this session. `PagedBootstrap` now precomputes
`treeCrumbs` via `BuildAncestorBreadcrumb(savedFocus)` and, when
restoreFromSelection + tree mode + non-empty chain, sets cursor =
`treeCrumbs[last]` (topmost ancestor folder).  SQL predicate becomes
`<= (isFolder=1, …)` which includes ALL folders + non-folders up to
chain[last] in the first batch — the auto-expand walker is guaranteed
to find chain head at depth 0 even when top-level folder count
exceeds viewport.  The same `treeCrumbs` array is reused by the
walker below without a second `BuildAncestorBreadcrumb` round-trip.

### TabularSection paging — landed 2026-05-07 (late)

Substantial second arc tackling RAM-backed paged TabularSection.
Eight distinct issues surfaced in cascade; closed end-to-end:

1. **StartEditing landed editor at the bottom of the form.**
   `ItemToRowJob::operator()` (datavgen.cpp:3232) compared
   `node->GetItem() == m_item` which after the refcount-aware
   ibDataViewItem refactor dispatches to
   `ibDataViewObject::IsEqualTo` — value-equality on
   `ibValueTableRow::m_nodeValues`.  For freshly created
   TabularSection rows with default / empty values that returns
   true across unrelated rows, so the walker stopped at the first
   value-equal sibling (typically `m_root.children[0]`).
   `GetItemRect` then computed `lineStart=0`, editor opened at
   y=0 of the data window — visually at the top of the table area,
   which on a doc form sits near the bottom.  Fix: walker compares
   by pointer identity (`GetID() == GetID()`) — instance lookup,
   not value equality.

2. **Container marker on folder rows disappeared after refresh.**
   `PagedBootstrap` had `if (!listMode && items[i].IsContainer())
   node->SetHasChildren(true);` while `OnPagedFetchForwardResult` /
   `OnPagedFetchBackwardResult` called `SetHasChildren` unconditional.
   Asymmetry: bootstrap rows lost the folder icon in List mode while
   fwd/bwd rows kept it.  Fix: drop the `!listMode` gate in
   bootstrap; symmetric with result handlers.

3. **Cell edit triggered full tree wipe + refetch (flicker).**
   `ibDataViewCtrl::DoItemChanged` for paged models always called
   `SchedulePagedRefresh(item)` — heavy hammer.  Fix: gate on
   `Features::RamFetch` — RAM-backed falls through to non-paged
   narrow path (`RefreshRow(GetRowByItem(item))` + value-changed
   event).  DB-backed keeps the full refresh because cursor needs
   resync against SQL.  Same gate applied to `ItemAdded` and
   `ItemDeleted`.

4. **Filter / sort silently no-op'd on RAM-backed tables.**
   `ibValueModelRamTableBase::GetFirstFetch / Next / Prev` sliced
   `m_nodeValues` raw — filter row and sort order were ignored.
   Fix: new `BuildVisibleView()` helper builds a filtered + sorted
   view of `m_nodeValues` per fetch (RAM tables small enough that
   per-call rebuild is cheap), Get*Fetch slices that view by
   anchor + count.  `RefreshTabularSection()` collapsed to a
   single `NotifyReset()` — flag-toggle path via `Show(item, vis)`
   and the per-row HIDDEN flag intermediate are gone.

5. **Add / Copy inserted at the active row's index, pushing it down.**
   User on visible row 3 + Add → new row replaced position 3,
   active row moved to 4 — confusing.  Fix: `AddValue` /
   `CopyValue` now insert at `row + 1` (after the active row);
   ItemAdded handler's `Select(item)` puts focus on the new row.

6. **Selection didn't move on Add / Delete.**
   Narrow `ItemAdded` / `ItemDeleted` only updated index counters
   via `m_selection.OnItemsInserted` / `OnItemsDeleted` — focus
   stayed on whatever was selected before.  Fix: call public
   `Select(item)` (which already does ExpandAncestors + single-sel
   UnselectAllRows + SelectRow + ChangeCurrentRow) for the new row
   in `ItemAdded`; for `ItemDeleted` move focus UP (`itemRow - 1`)
   so the row above the deleted one becomes active.

7. **Row-number column showed insertion-order indices, not display
   positions.**  `GetValueByRow` for the number-line column did
   `GetRow(row) + 1` where `GetRow` returned m_nodeValues raw
   index — chaotic on filter/sort and on middle-inserts.
   `GetRow` and `GetItem` are kept as RAW accessors (save loop
   iterates `0 .. GetRowCount()-1` over m_nodeValues directly);
   number-line resolution now happens inline in
   `GetValueByRow` via `BuildVisibleView` so the user sees
   1, 2, 3 … N in render order regardless of underlying state.

8. **Focus reset to row 0 after refresh on tables with many
   default-valued rows.**  `PagedBootstrap`'s post-fetch focus-scan
   (`items[i] == savedFocus`) hit the same IsEqualTo false-positive
   trap as ItemToRowJob — items[0] was always "equal" to
   savedFocus when both had default values, so currentRow always
   landed on 0.  Fix: capture `m_pagedRestoreFocusRow = (long)
   m_currentRow` at refresh time; in bootstrap, when model has
   `Features::RamFetch` and the captured row is in range, restore
   focus by index directly — no value-eq scan.  DB-backed paged
   keeps the IsEqualTo scan (rows have unique GUIDs, no false
   positive).

### Pointer-identity audit

The recurring root cause across (1), (3), (5), (6), (8) was the
same: ibDataViewItem `operator==` after the refcount-aware refactor
dispatches to `ibDataViewObject::IsEqualTo` (value-based, useful
for selection survival across re-fetch on Catalog/Document lists
where rows have unique GUIDs), but TabularSection rows have
`m_nodeValues` value-equality that's too permissive for default-
valued rows.  Walker / insertion / deletion paths need INSTANCE
identity, not value equality.  Audit pass split the use sites:

* **Pointer-identity** (`GetID() == GetID()`):
  - `ItemToRowJob::operator()` — generic GetRowByItem walker.
  - `ibDataViewTreeNode::FindChildByItem` — sibling lookup in
    InsertChild / Delete paths.
  - `FindNode` parent-chain walker — used by non-paged
    DoItemChanged.
  - Narrow `ItemAdded` posInModel reverse-search — replaces
    `modelSiblings.Index(item, fromEnd=true)` which used value-eq.
  - `ItemDeleted` itemNode loop — finds the just-deleted node.

* **Value-equality** (`IsEqualTo` via operator==):
  - Tree auto-expand walker (`PagedBootstrap`'s `treeCrumbs` loop)
    — ancItem comes from a separate `BuildAncestorBreadcrumb`
    fetch with different pointer than current m_root.children;
    value-eq via m_objGuid is what makes cross-fetch matching work.
  - `findVisibleRow` lambda — savedFocus is the pre-refresh
    pointer, m_root.children has post-fetch fresh pointers; same
    cross-fetch matching by m_objGuid.

## Decisions to remember

* **Always check what the column actually stores before claiming
  a driver "can't bind reliably".** The FB binary CHAR(16) comment
  in `objectListQuery.cpp` carried a wrong premise (CHAR(16) vs
  BINARY(20)) and a wrong remedy (skip SQL filter, do C++ match)
  for two months. Diagnostic that helped: `GetResultBlob` (full
  bytes, no string truncation) + literal `WHERE col = X'<hex>'`
  probe. If literal matches and bind-via-param doesn't, bind path
  is broken; if literal also fails, the in-memory target is wrong.
* **`ibReference` always carries the table's metaID, even for
  empty references.** The save path's `SetParamNull` doesn't
  result in NULL/empty rows on BINARY columns — it round-trips
  as `<zero guid><table metaID LE>`. Anyone doing parent-ref SQL
  filtering must build the same shape, not all-zero.
* **FB driver's `SetParamBlob` was a silent no-op for SQL_VARYING
  parameters.** Future bug-source. Patched.
* **`ibDataViewItem::operator==` is value-equality, not pointer
  identity.**  After the refcount-aware refactor it dispatches to
  `ibDataViewObject::IsEqualTo`.  That's correct for cross-fetch
  selection survival on Catalog/Document lists (unique-GUID
  rows).  For walkers / insertion / deletion paths that need to
  locate a specific instance, compare by `GetID()` directly —
  IsEqualTo on TabularSection's value-map produces false positives
  on default-valued rows.  Five callsites audited; the two that
  legitimately need value-eq are documented with comments.
* **GetRow / GetItem on RAM-backed paged report RAW m_nodeValues
  positions, not view positions.**  Save / iteration code uses
  `for (long i = 0; i < GetRowCount(); ++i) GetItem(i)` and needs
  every row including filtered-out ones.  Display-position queries
  (number-line column) re-resolve through `BuildVisibleView`
  inline.  Don't conflate the two semantics.
* **Focus restoration across paged refresh: raw row index for
  RAM, IsEqualTo scan for DB.**  RAM-backed tables preserve order
  through save/reload (line-number ORDER BY), so capturing
  `m_currentRow` at refresh time and restoring by index is more
  reliable than value-eq scan.  DB-backed cursor pagination has
  unique GUIDs so IsEqualTo is correct there.
