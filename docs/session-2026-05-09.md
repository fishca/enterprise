Catch-up day after the 2026-05-08 datavgen split.  No new feature work
— a series of regressions surfaced when bringing the working copy back
under control after the diagnostic-log strip, plus a couple of

Build clean Debug/x86, validated through `launcher.exe` on

## Regressions closed

### 1. Cold-open empty-table — `m_pagedNeedsBootstrap` reset stuck under dangling-if

`PagedBootstrap` ended with:

```cpp
if (prevHasFwd != m_pagedHasMoreFwd)
        if (prevHasBwd != m_pagedHasMoreBwd)
        m_pagedNeedsBootstrap   = false;
```

Strip pass earlier in the day removed the diagnostic-log bodies of two
nested `if`s, leaving them brace-less.  C parsing rule then made
`m_pagedNeedsBootstrap = false;` the body of the *inner* `if` →
reset only fires when both `prevHas*` differ.  On cold-open both
stayed equal → flag stuck `true` → every `OnInternalIdle` re-armed
the bootstrap → infinite wipe-fetch-wipe loop, table flickered to
empty.

Fix: reset is unconditional, the two `if`s carry braces so future
strip can't repeat the trap.  See `datavgen.paged.cpp` end of
`PagedBootstrap`.

Added a static-analysis guard (one-shot Python script) that scans
stripped files for control-flow statements where the next non-blank
line is `}` or `else` — same pattern catcher.  No remaining hits.

### 2. Sort loses focus when row resorts beyond the first batch

Repro: select a row near the top, click a column header that pushes
the row past the initial fetch batch (e.g. position 5 → 90 in a
catalog with `WHERE` filter excluding the row from the head).
PagedBootstrap fetched `cursor = m_pagedRestoreAnchor` (empty under
sort because `skipCapture` had wiped it), got the first 20 rows of the
new ordering, IsEqualTo scan missed the saved focus.

Fix lives in two places:

* `PagedRefresh` capture — under `skipCapture` (sort path), the
  anchor / selection are still wiped (cursor invalidated), but
  `m_pagedRestoreFocus` and `m_pagedRestoreFocusRow` are now captured
  from `m_currentRow`.  Drill UP / INTO also use `skipCapture`, but
  there old focus is from a different parent and the IsEqualTo scan
  silently misses — no destructive overwrite of a still-valid sort
  capture on a concurrent path.
* `PagedBootstrap` — when the non-selection path has an empty anchor
  (sort), fall back to focus-as-cursor so the post-sort fetch
  positions around the previously-selected row.  Plain-Refresh path
  still uses anchor first (preserves viewport).

`currentFocus` capture moved out of the `if (!skipCapture)` gate —
without that the sort path saw an empty `currentFocus` regardless of
`m_currentRow`, and the focus capture above was a no-op.

### 3. Add to empty folder didn't activate the new row

`Select(item)` on a paged model stamps `m_pagedRestoreSelection` via
`SetPagedRestoreSelection` but did not stamp `m_pagedRestoreFocus`.
PagedBootstrap then read `savedFocus = m_pagedRestoreFocus` (empty)
and skipped the focus-restore loop entirely — even though the new
row sat in the fetched batch.

Fixes:

* `SetPagedRestoreSelection(item)` now also stamps
  `m_pagedRestoreFocus` and forces `m_pagedRestoreFocusWasSelected =
  true` (programmatic Select always means "really select this row";
  the wasSelected flag had previously been captured from
  `!m_selection.IsEmpty()` *before* Select stamped, so post-Add to a
  list with no prior selection landed `wasSelected = false` and the
  focus-restore loop did `ChangeCurrentRow` only, skipping
  `SelectItem`).
* `savedFocus` in PagedBootstrap is now
  `m_pagedRestoreFocus.IsOk() ? m_pagedRestoreFocus : m_pagedRestoreSelection`
  — defence in depth, so post-Add scenarios that bypass
  `SetPagedRestoreSelection` still work.

### 4. `tableBox` AssociateModel reset wiped the just-applied current line

Tab rebuild after `createdValue` notification destroys the previous
`ibTableViewCtrl` and creates a new one with the same model.  The
new control's `dataViewOldModel == nullptr` → the legacy
`m_tableCurrentLine.Reset()` fired, wiping the line `ApplyCurrentLine`
on the previous control had just set.  Symptom: just-added row
appeared but didn't get the focus.

Fix: distinguish "fresh control attaching to existing model" (carry
`m_tableCurrentLine` forward + re-route through `Select(item)` so the
new control's bootstrap-restore channel picks up the line) from "real
model swap" (legitimately reset).

### 5. Hierarchical drill flicker — async PagedRefresh outside freeze

`SetTopParent` (drill UP / INTO) wrapped `Cleared() + PagedBootstrap`
in `ScopedPagedFreeze`, expecting the sync sequence to land inside
the freeze.  But `Cleared()` for paged routes through
`SchedulePagedRefresh` which dispatches `PagedRefresh` via
`SubmitFetchAsync` — async on web, inline on desktop.  If the inline
path didn't execute fully synchronously (e.g. dispatch-to-CallAfter
on some build configurations), `m_pagedNeedsBootstrap` stayed false
inside the if-check, the sync `PagedBootstrap` call was skipped, the
ScopedPagedFreeze thawed against a still-old tree → user saw the
old folder briefly before the eventual idle bootstrap fetched the
new one.

Fix: replace `Cleared()` with a direct `PagedRefresh()` call in both
drill branches.  `PagedRefresh` is sync, arms
`m_pagedNeedsBootstrap`, and the subsequent `if` actually fires
`PagedBootstrap` inside the freeze.

Bonus fix in the same edit: `ScopedPagedFreeze` already freezes
`m_tableAreaWin`; `PagedBootstrap`'s own
`m_pagedFrozenForBootstrap` freeze adds a second layer that survives
past the outer scope's `~Thaw` → `m_table` stays frozen at depth 1
until the next `OnInternalIdle` thaws the bootstrap-owned freeze.
For the drill path we set `m_pagedFrozenForBootstrap = true` before
calling `PagedBootstrap` (suppressing its inner freeze) and restore
the old value after, so the single outer freeze cleanly Thaws when
the scope exits.

### 6. Hierarchical breadcrumb — clicking ancestor crumb duplicated it

`SetTopParent` only treated `item == GetTopParentItem()` (chain[0]) as
drill-UP.  Clicking any deeper crumb (chain[1]…) fell through to the
drill-INTO branch and `Insert(item, 0)` appended the clicked crumb in
front of itself in the chain.  User saw "click on top of the
breadcrumb → it appears at position 3 too".

Fix: scan the chain for `item`; if found at index `i`, pop
`chain[0..i]` so that crumb's parent becomes the new top.  Drill-INTO
branch handles only items not in the chain.

### 7. Filter + Add row that misses the filter → empty table

Cursor-mode bootstrap with `cursor = newRow` and a SQL filter that
excludes the new row returns 0 rows (cursor predicate `>= newRow's
sortValues` excludes everything in the filtered scope).  Result:
table fully blank.

Fix: empty-fetch fallback in `PagedBootstrap`.  When the cursor-mode
fetch returns `n == 0` and the cursor was non-empty, retry with an
empty cursor → user sees the filtered top batch instead of nothing.

### 8. `m_eventOnAddRow` fired twice

Single user "Add" via the GUI:
* tableBoxEvent.cpp `OnItemStartAdding` fires `OnAddRow` (notifier
  →`ItemAppended` for RAM-paged TabularSection).
* `tableBox.cpp` OnUpdated/`fromCreated` fires `OnAddRow` again
  (DB-paged catalog post-save, after `ResolveLineByValue`).

Removed the OnUpdated firing — `OnItemStartAdding` is the single
source of truth.  `fromCreated` flag dropped.  DB-paged catalog Add
notification flows through the model's notifier (notifier
→`ItemAppended` if needed, otherwise no `OnAddRow` is the right
behaviour for a refetch-driven add).

## Cosmetic / unrelated

### 9. Cell-mode subtle row hint

User feedback — in `ibDataViewSelectCell` mode the focused cell got a
strong native selection rect, but the rest of the row had no cue at
all.  Hard to track which row the active cell belongs to.

Added a 15% blend of system-highlight over the row background as a

back to 15% (the 4% cut was too subtle to register).

### 10. Sort + filter types moved tableInfo.h → tableView.h

`ibSortOrder` / `ibSortData` / `ibSortModel` / `ibFilterRow` /
`ibComparisonType` lived in `tableInfo.h`.  Datavgen needed full
`ibSortOrder` for the header-arrow update path
(`SyncColumnArrowsFromModel`) and would have had to pull in
`tableInfo.h` — but `tableInfo.h` includes the value-subsystem
headers, which isn't appropriate for a frontend control.

Moved both groups to `tableView.h` (the view-layer header that
datavgen already depends on).  `tableView.h` now pulls
`backend/system/value/valueType.h` for `ibValue` / `ibTypeDescription`
/ `ibGuid`; no cycle (value subsystem doesn't include `tableView.h`).

## Decisions to remember

* **Strip script must clean up empty if-bodies.** Removing
  `ibPagingLog(...)` calls inside brace-less `if/else` left dangling
  control-flow statements that captured the next non-log statement
  as their body.  Future diagnostic-strip should flag and patch
  those (one fix this session was a multi-day-undetected
  `m_pagedNeedsBootstrap = false` becoming the body of a nested
  `if`).  Static check: pattern matcher for control statement on
  its own line followed by `}` or `else` on the next non-blank line.
* **Programmatic Select implies wasSelected = true.**  The
  PagedRefresh capture path reads `!m_selection.IsEmpty()` for
  wasSelected — a fine signal for plain Refresh / sort, but
  post-Add to an unselected list lands wasSelected=false because
  Refresh runs *before* Select stamps the new row.
  `SetPagedRestoreSelection` now overrides wasSelected to true so
  the focus-restore loop applies `SelectItem`, not just
  `ChangeCurrentRow`.
* **Cursor-mode fetch needs an empty-cursor fallback.**  Filter
  + Add of an out-of-filter row used to leave the table blank
  because the cursor predicate against the new row's sort values
  excluded everything.  Always retry with empty cursor when the
  primary fetch returns 0 — yields the filtered top batch.
* **Drill freeze must own the bootstrap synchronously.**  Async
  `Cleared() → SchedulePagedRefresh` left the freeze guarding only
  the metadata mutation, with the actual fetch happening later
  outside the scope.  Direct `PagedRefresh()` keeps the wipe + fetch
  sequence inside the freeze; the `m_pagedFrozenForBootstrap`
  override prevents the inner freeze from outliving the outer scope.
* **Breadcrumb click resolves to chain index, not equality with
  chain[0].**  Old code only handled the current top; clicking
  ancestor crumbs duplicated them.  Linear scan of the chain is
  fine — chain depth is rarely above 4-5.
