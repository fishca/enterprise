## 1. Iterator subsystem — phases 1–5

The old surface had four virtuals per iterable: `HasIterator()`,
`GetIteratorEmpty()`, `GetIteratorAt(idx)`, `GetIteratorCount()`. The
runtime drove iteration through an inline `ibValueIterator` that held
`ibValue& m_ownerValue` and walked indices. Several latent bugs and
no path to native cursor-paginated DB iteration.

### Phase 1 — cursor API + Break-leak fix

New abstract base in `value.h`:

```cpp
class ibValueIteratorState {
public:
    virtual ~ibValueIteratorState() = default;
    virtual bool MoveNext(ibValue& current) = 0;
    virtual void Reset() = 0;
    virtual bool PeekSample(ibValue& current) const { return false; }
};
```

`ibValue::CreateIterator()` returns `std::shared_ptr<ibValueIteratorState>`.
Default delegates through TYPE_REFFER chain to the underlying object;
returns null when not iterable.

`ibValueIterator` runtime holder rewritten: holds the state plus a
`m_hotFromNextIter` flag. `OPER_NEXT_ITER` sets the flag, `OPER_FOREACH`
consumes it. Re-entry into FOREACH with the flag clear (Break-leak from
an outer-loop iteration) drops the stale state and rebuilds from
`variable2` — which is the just-LET'd `@in_` slot, so a re-evaluated
iterable picks up correctly.

Off-by-one fixes (`>` → `>=`) in four spots: `tableInfo.h`,
`valueTable.h`, `tabularSection.h`, `commonObject.h`.

Removed `s_defaultOwner` static and the legacy `GetCurrentValue /
NextIterator / ResetIterator` accessors.

### Phase 2 — RAM-paged filter+sort consistency

`ibValueModel::CreateIterator()` overrides return a paged-cursor state
(`ibValueModelPagedIteratorState` in `tableInfo.cpp`) that drives the
model's `GetFirstFetch / GetNextFetch` in 64-row batches. RAM-paged
children (TabularSection / valueTable / RecordSet) automatically
inherit filter+sort consistency with the GUI because their Get*Fetch
slices `BuildVisibleView`.

Detection: `Features::RamFetch | Features::DbFetch` set, `Tree` clear.
Tree models fall through to the base default (no flat-walk over a tree
— would surprise users).

### Phase 3 — DB-paged lists script-iterable

Catalog / Document / Register lists become script-iterable for free
through the same `ibValueModel::CreateIterator` path. `EnumRef` got a
missing `GetFeatures` override (DbFetch flag) so it slots into the
same machinery. No per-class iterator-state code.

### Phase 4 — legacy API cleanup

`HasIterator / GetIteratorAt / GetIteratorCount` removed from `ibValue`
base API and all overriders (~13 places). Each non-Model iterable
class (`ibValueArray`, `ibValueContainer/Structure`,
`ibValueModelColumnCollection`, three record-prop walkers in
`commonObject.h`, `ibValueForm::ibValueFormCollectionControl`,
`ibValueFrame::ibValueEventContainer`) got its own inline
`CreateIterator()` override walking its native storage directly. The
former `IndexedIteratorState` wrapper that bridged the old API to the
new one is gone — there's nothing left to bridge.

`RefreshTabularSection()` deleted (zero callers post-2026-05-07
`NotifyReset()` collapse).

### Phase 5 — PeekSample + shared_ptr

`GetIteratorEmpty()` was the only legacy survivor (used by IntelliSense
type-hint). Moved into the iterator state itself as
`PeekSample(ibValue&) const` — symmetric with `MoveNext`'s out-param
shape. IntelliSense's call (`codeEditorInterpreter.cpp:1265-area`) now
goes `state = paramObject.CreateIterator(); state->PeekSample(skel);`.

Model children's typed-skeleton factory moved from `GetIteratorEmpty`
to a fresh `ibValueModel::GetEmptyRow()` virtual; `ibValueModelPaged
IteratorState::PeekSample` calls it and treats `TYPE_EMPTY` as "no
hint".

`m_state` in `ibValueIterator` switched from raw `ibValueIteratorState*
+ delete in dtor` to `std::shared_ptr<ibValueIteratorState>` per
project convention (`feedback_prefer_shared_ptr`). All `CreateIterator`
overrides return shared_ptr / `make_shared`.

## 2. Shutdown deadlock — registry ↔ ThreadBody / debug listener

Diagnosed from a real `зависший.dmp` via cdb. Two coupled hangs.

### Listener spin

`ibDebuggerServerConnection::EntryClient` (`debugServer.cpp:684`) cycles
through `wxSocketServer::WaitForAccept(0, waitDebuggerTimeout)` with
TestDestroy poll between ticks. `wxThread::Delete()` sets the destroy
flag; the loop exits within one tick. But on a crash path
`ibDebuggerServer::Destroy()` was never called, so the flag was never
set. The wxTHREAD_JOINABLE listener kept the process alive.

There is also an `Accept(true)` branch in the same loop (taken when
`m_waitConnection == true`) that blocks indefinitely without polling
TestDestroy. Even a correct Delete + Wait wouldn't break that one.

Fix in `ShutdownServer`: destroy `m_socketServer` before `thread->Wait()`.
WaitForAccept / Accept return immediately on a destroyed listener,
TestDestroy fires on the next iteration, the thread exits cleanly. The
worker's own dtor / OnKill paths null the same pointer; the duplicate
Destroy is a no-op.

### Registry / main thread mutex chain

Single backend `.join()` callsite — `ibSessionRegistry::Stop` joining
`m_thread` (the ThreadBody). The original Stop() submitted Remove
requests for every owned session before joining; ThreadBody picked them
up via `Submit + Drain` and ran `ProcessRemove` from the registry
thread. Each ProcessRemove fires `NotifyDisconnect` listeners,
`DetachRuntime` on the module manager, etc.

`BeforeStart` (`moduleManagerEvent.cpp:37`) takes a
`lock_guard<m_runtimeMutex>` for the ENTIRE `CallAsProc("beforeStart")`
duration. SEH dispatch does NOT unwind C++ frames before the filter
runs, so a script crash inside BeforeStart leaves the lock_guard held
while the SEH chain runs OnFatalException → DestroyAppDataEnv →
~ibApplicationData → registry.Stop() → m_thread.join(). ThreadBody is
mid-ProcessRemove → DetachRuntime → blocked on the same m_runtimeMutex.
Deadlock.

Two-step fix:

A. **Inline drain in Stop().** Set m_stop, notify, join — then drain
   Removes inline on main, NOT through ThreadBody. ThreadBody's queue
   stays empty, it sees m_stop and exits. Removes the cross-thread
   mutex chase entirely on the normal path.

B. **Timed join + detach backstop.** Even after (A), a Remove that was
   already in flight when Stop ran can leave ThreadBody mid-ProcessRemove.
   Stop now polls `m_threadAlive` with a 2-second deadline; on timeout
   it calls `m_thread.detach()` and skips the inline drain (which would
   itself deadlock for the same reason). Process exits, OS reaps the
   thread.

The detach branch logs a warning. If we ever see it in normal use
that's a real bug; on crash paths it's the expected escape hatch.

## 3. IntelliSense — four small items

### Item 1: anonymous functions in expressions

`var f = Function(x) ... EndFunction;` and the CES counterpart
`var f = Function(x) { ... };` weren't expressions to the IntelliSense
parser — `GetExpression` had no KEY_FUNCTION/KEY_PROCEDURE branch.

Added a depth-tracked skip that swallows the signature `(...)` and the
body, returning a typed-empty `FUNCTION` variable. Body parsing is
deliberately skipped (a full pass would need a child `m_activeContext`
push/pop); a future iteration can expose lambda locals to autocomplete.

### Item 2: preprocessor directives

`#define X`, `#region X`, `#ifdef X` etc. — the lexer was dropping the
keyword via `continue` in both PrepareLexem variants but keeping the
following identifier as a free token. The parser then tried to interpret
that ident as an expression and fell over.

Fix: when a directive keyword is seen, record `directiveLine =
m_currentLine` and drop every subsequent token while we're still on
that line. Applied at all four emit sites (full + incremental tokenizer,
keyword-path + CONSTANT-path each).

### Item 3: lambda eval crash logging

User reported a silent process death when evaluating an expression in
the debugger watch panel from inside a paused lambda. Hard SEH; the
shutdown deadlock fix from §2 made the death stop hanging the process,
which was a prerequisite for collecting a clean dump.

Added `lambda.log` file-based logging in CWD. `wxLogDebug` alone
routes only to OutputDebugString; users without DebugView never saw
the messages. The new helper writes a timestamped line per call:

```
[09:14:18.123] [lambda] OPER_LFUNC materialise: bc=0x… funcIdx=N endIp=M lambdaName=<lambda@N>
[09:14:18.124] [lambda] OPER_CALL_VAL: <lambda@N> callerArgs=K lambdaParams=M lambdaVars=V
[09:14:18.125] [lambda] Execute begin: lambda=<lambda@N> parentBc=0x… …
[09:14:18.125] [lambda] body ip=3 op=33 p1=(a=0,i=0) p2=(a=-1,i=-1) p3=(a=0,i=0) p4=(a=0,i=0)
[09:14:18.125] [lambda] body ip=4 op=1  p1=(a=-3,i=1) p2=(a=1000,i=1) …
[09:14:18.125] [lambda] body ip=5 op=55 p1=(a=-3,i=2) … method='Message'
[09:14:18.140] [lambda] dispatch ip=3 op=33
[09:14:18.140] [lambda] dispatch ip=4 op=1
   ← (no further entries: process died at ip=5)
```

Per-opcode dispatch trace is gated on
`pContext->m_currentFunction->IsLambda()` so the log doesn't get
flooded by named-function execution.

### Item 4: CES syntax support

The IntelliSense's `CompileBlock` and statement parsers (`CompileIf` /
`CompileWhile` / `CompileFor` / `CompileForeach` / `CompileException`
/ `CompileFunction`) were VBS-only. CES code in the editor broke
autocomplete past the first control structure.

Mirrored the backend's `ibCompileCode`'s CES patterns:

| Construct | VBS | CES |
|---|---|---|
| Block | keyword-fenced (EndIf / EndDo / EndFunction / …) | `{ … }` |
| `if` | `If c Then … EndIf` | `if (c) { … }` or `if (c) stmt;` |
| `while` | `While c Do … EndDo` | `while (c) { … }` |
| `for` | `For i = 1 To N Do … EndDo` | `for (i = 1 To N) { … }` |
| `for each` | `For Each x In c Do … EndDo` | `for each (x In c) { … }` |
| `try` | `Try … Except … EndTry` | `try { … } except { … }` |
| `function` | `Function f(…) … EndFunction` | `Function f(…) { … }` |
| inline lambda | `Function(x) … EndFunction` | `Function(x) { … }` |
| `;` | terminates a block | statement separator |

Brace-less single-statement form `if (cond) Message();` was an
afterthought — the backend uses `m_numReturn == RETURN_BLOCK` to know
"single statement only"; IntelliSense doesn't have that context kind,
so a local `singleStmtMode + parsedOne` flag pair handles it. Set
`parsedOne = true` after each completed statement; if singleStmtMode
exit on the next loop turn.

`KEY_RETURN` correctly handles `;` at expression-required position;
the existing `if (IsNextDelimeter(';')) return false;` was already
right for both styles (CES uses `;` to end the statement).

The mode is read from `ibCompileCode::GetCodeStyle()` — process-global
flag set by metadata loader / codeRunner UI dropdown.

## 4. Lambda eval-in-lambda — root cause + temporary guard

Diagnosed via the new logs + a fresh dump.

The crash chain is:

1. Lambda body executes (CALL_VAL → ibValueFunction::Execute → runtime
   ProcUnit Execute with `m_pByteCode = m_parentBc`).
2. Breakpoint fires; `EnterDebugger` → `DoDebugLoop` → user types
   `Catalogs.X` in watch.
3. `SendExpressions` → `ibProcUnit::Evaluate` → compiles a fresh eval
   bytecode via `ibCompileEval` against the current run context (the
   lambda's frame).
4. Eval ProcUnit's `m_pppArrayList` is laid out as
   `[0=own, 1=host=lambda, 2=host's-parent-module, …]` per
   `Evaluate`'s comment. For a lambda host, depth ≥ 2 is null —
   lambdas don't carry a parent-module-frame in pppArrayList; their
   parent walk lives on the bytecode side via `m_parent`.
5. Eval bytecode's `Catalogs` resolution emits a slot in the depth=2+
   range. ResolveWrite/ResolveRead deref `pppArrayList[depth][idx]`
   with `pppArrayList[depth] == nullptr`. Hard SEH.

### Temporary guard

`ResolveWrite` / `ResolveRead` got null-and-bounds checks (depth ≥ 0,
`pppArrayList != nullptr`, `pppArrayList[depth] != nullptr`,
`pppArrayList[depth][idx] != nullptr`). On any miss, throw
`ibBackendException("Outer frame not bound at depth N / idx K (eval
inside lambda?)")`. Watch panel surfaces a readable error; lambda
keeps running.

### Real fix — open

The eval ProcUnit needs to wire its `pppArrayList` to the lambda
runtime's chain instead of the host-frame chain. Two approaches sketched
in chat:

A. At `Evaluate` entry, detect "host is lambda" and substitute the
   pppArrayList with `[own, host=lambda, root_mm_frame_1,
   root_mm_frame_2, …]` (the runtime's chain that a normal lambda
   body opcode walks).
B. Compile eval bytecode with scope = lambda's parentBc; eval ProcUnit
   pulls the runtime's pppArrayList directly.

Either path is ~50–100 lines in the eval machinery. Deferred — the
guard is enough to keep the watch panel from killing the process, and
the SEH-via-shutdown problem is fixed independently.

## Decisions to remember

* **CES is the new default authoring mode.** Existing modules / configs
  preserve their stored Syntax property (so VBS-marked modules still
  parse VBS). Brand-new configurations land in CES.
* **Iterator API single source of truth: `CreateIterator()`.** Returns
  shared_ptr to a state with MoveNext / Reset / PeekSample. The legacy
  random-access surface (At/Count/Has/Empty) is gone from `ibValue`.
* **`GetEmptyRow()` on `ibValueModel`** — typed-skeleton factory used
  by the paged iterator's PeekSample. `TYPE_EMPTY` return = no
  IntelliSense hint; only TabularSection / valueTable / RecordSet
  override with real ReturnLine factories.
* **Lambda body always runs on the session's lambda runtime ProcUnit
  with `m_pByteCode` swapped to `m_parentBc`.** `ibValueFunction::Execute`
  guards with try/catch around the runtime->Execute call; on exception
  it logs the lambda name and `pContext->m_lCurLine` (last-attempted
  ip) before rethrowing.
* **Stop's job is to drop the queue, not to drain it.** The original
  shape (Submit Remove, then join) coupled main's shutdown to ThreadBody's
  in-flight work. Inline drain on main + timed-join + detach backstop
  is the canonical shape now.
* **Preprocessor directives consume the whole line in the IntelliSense
  lexer.** Same treatment as the comment-strip rule from earlier
  sessions — drop everything from the directive keyword to the next
  newline.

### 1. CES auto-indent in code editor

Soft single-statement `if (cond) stmt;` form (CES) was tracked
nowhere — fold parser only marks `{...}` braces, so body line of a
brace-less header had no fold flag and `PrepareTABs` stripped its
`\t` on Enter. Five rounds of fixes landed:

* `IsCESSingleStmtHeader(line)` detector — line starts with
  `if`/`while`/`for`, ends with `)`, no `{`, comment-tail trimmed.
* `softBodyExtra = 1` when prev (skipping blanks) is a soft header —
  added to every branch's `applyIndent` target so body indent is
  preserved on Enter.
* `currIsSoftHeader → foldLevel += 1 + softBodyExtra` for new-line
  indent; `currLineBlank && prevIsSoftHeader → foldLevel +=
  softBodyExtra` so an empty `\t|` line doesn't drop back to col 0
  on Enter (still inside the soft body region pre-body).
* CES `applyIndent` is **no-shrink** — fold parser's foldLevel is
  often less than user's manual indent (no fold between soft scopes)
  so target=foldLevel would strip; only fill the deficit, never
  trim. VBS keeps strict-match (its fold vector is keyword-fenced
  and accurate).
* `IsLineJustOpenBrace(line)` + `currLineIsBraceOpener` — Allman
  `{` on its own line "consumes" the soft-body bump. Without this
  the `{` would land at body indent (one tab too deep) on Enter.

`OnCharAdded` handler bound to `wxEVT_STC_CHARADDED`:

* **`}` typed on a whitespace-only line** → walk back depth-tracked
  through raw bytes (wxSTC's `BraceMatch` returns -1 because our
  custom styler doesn't tag braces), find matching `{`, set line
  indentation to opener's. Auto-aligns Allman closers.
* **`{` typed on a whitespace-only line** (CES only) → walk back
  past blanks looking for a soft header; if found, dedent current
  line to header's indent. Allman dedent under `if (cond) \n {`.

### 2. Brace highlight + gutter cosmetics

Brace highlight via `wxEVT_STC_UPDATEUI` — manual matcher (same
brace-balance walk as `}` align), supports `{}` `()` `[]`. Style:
red foreground only, no bold or background — font metrics stay
stable while caret moves over braces.

`SetFontColorSettings` order fix: `StyleSetFont(STYLE_DEFAULT, font)`
+ fg/bg now happen **before** `StyleClearAll()` so the cascade
propagates the user's font (and point size) to every style index.
Without this, untouched styles (annotations, indent guides, callTip,
control chars, BRACELIGHT/BRACEBAD) kept the OS-default font and
zoom +1pt was applied to a different base — visually choppy /
non-proportional zoom.

Gutter gradient: line-number margin darkest, breakpoint medium,
fold lightest. `wxSTC_STYLE_LINENUMBER` bg = `#E8E8E8`, digits
`#A0A0A0`. `SetMarginBackground(BREAKPOINT, #F0F0F0)`,
`SetMarginBackground(FOLDING, #F6F6F6)`,
`SetFoldMarginColour(true, #F6F6F6)` + Hi colour. Margins read
as one continuous band fading into the white text area.

`MarginBackground` calls go through `SetMarginBackground(margin,
color)` (wxSTC API) — earlier `StyleSetBackground(margin_id, ...)`
on the breakpoint margin was a bug (margin id treated as style
index — set style 1 = wxSTC_C_DEFAULT to white).

BRACELIGHT/BRACEBAD foreground reapplied at the end of
`SetFontColorSettings` (after `StyleClearAll`) — otherwise the
ctor-time setup wiped on every settings refresh.

### 3. Lambda eval real fix — `m_pppArrayList` splice

Morning's guard converted SEH into ibBackendException. Real fix
landed in PM:

`ibProcUnit::CompileExpression` else branch (non-eval-of-eval host):
when `pRunContext->m_currentFunction->IsLambda()`, splice the
session's lambda shim's `m_pppArrayList[1..]` into eval's
`m_pppArrayList[2..]`. Eval has one extra own-frame layer above
shim (eval → shim → rootPu → …), so offset is `+1`.

```cpp
if (pRunContext->m_currentFunction != nullptr
    && pRunContext->m_currentFunction->IsLambda())
{
    ibProcUnit* const shim = pRunContext->m_procUnit;
    if (shim != nullptr && shim->m_pppArrayList != nullptr) {
        const unsigned int shimSize = shim->GetParentCount() + 2;
        const unsigned int evalSize = GetParentCount() + 2;
        for (unsigned int k = 2; k < evalSize; ++k) {
            const unsigned int srcIdx = k - 1;
            if (srcIdx < shimSize)
                m_pppArrayList[k] = shim->m_pppArrayList[srcIdx];
        }
    }
}
```

**Root cause**: lambda shim is a dispatcher, its `m_cCurContext` is
empty (frames live directly in `m_pppArrayList`, set by
`ibSession::CompileRoot`). The outer `Execute` populates eval's
`[k+2]` from `parent->m_cCurContext.m_pRefLocVars` — for the shim
that's null, hence depth ≥ 2 lands in null and SEH'd before the
guard.

After splice: depth 2 in eval → shim's `[1]` (rootPu frame), depth
3 → shim's `[2]` (= rootPu, duplicate but correct), depth 4+ →
common-module / parent frames. Catalogs / Documents / Manager
resolve through eval as expected.

Guard in `ResolveWrite/Read` stays as defensive — if splice ever
fails or shim doesn't supply something, watch panel shows readable
error instead of segfault.

### 4. IntelliSense in CES + lambda body

Reported one after another: `CompileModule` returned false on every
keystroke, autocomplete dead.

Three independent fixes landed:

**(a) `CompileDeclaration` consumes initializer.** CES `var x =
Catalogs.Items;` left the chain expression in the lexem stream;
outer `CompileBlock` then read it as a fresh statement and failed
on the `.` (no leading identifier state). Now calls `GetExpression()`
after `=` and captures inferred type into `AddVariable`. VBS uses
`Перем x;` without `=` so the path doesn't fire there.

**(b) `CompileBlock(bool allowSingleStmt = true)`.** Module-level
body in CES has no `{` opener but is multi-statement. The original
heuristic `if (IsCES() && !'{') singleStmtMode = true` was wrong
for module body — after the first `;` the block exited, leaving
the rest as trailing tokens, `cursor+1 < size-1 → return false`.
`CompileModule` now calls `CompileBlock(/*allowSingleStmt=*/false)`.
Control-statement callsites (`CompileIf` / `CompileWhile` /
`CompileFor` / lambda body) keep the default `true` for brace-less
single-stmt support.

**(c) Inline-lambda in `GetExpression`.** Was a depth-tracked body
skip → autocomplete inside lambda body got the enclosing function's
scope (or nothing). Now pushes a child `ibPrecompileContext` with
parent = root, registers params via `AddVariable`, calls
`CompileBlock` to walk the body, sets `m_cursorContext =
m_activeContext` if caret falls in, restores `prevActive` on exit.
Catalogs / Manager / Documents now visible via parent = root chain;
lambda's params + locals visible via the pushed context. Mirrors
runtime discipline (no enclosing-function scope leak).

### Decisions to remember (PM)

* **CES = no-shrink indent rewrite.** Fold parser only tracks
  braces; manual indent in soft-body regions or hand-aligned closers
  must be preserved. VBS stays strict-match.
* **`{` consumes soft-body bump (Allman).** When a CES single-stmt
  header is followed by a brace-only line, the `{` lands at header
  indent — body inside `{...}` is at +1 over the brace. `OnCharAdded
  '{'` enforces this on type, `IsLineJustOpenBrace` keeps PrepareTABs
  from undoing it.
* **`}` align uses manual brace-walk, not `BraceMatch`.** wxSTC's
  built-in needs lexer-side brace styling; our custom styler doesn't
  tag braces.
* **Lambda shim's frame chain lives in `m_pppArrayList` directly,
  not in `m_cCurContext`.** Set by `ibSession::CompileRoot` once
  per session. Eval inside lambda must splice this chain into its
  own array — the default `Execute` setup walks parents'
  `m_cCurContext` which is empty for the shim.
* **`CompileDeclaration` consumes the initializer expression.** CES
  `var x = expr;` requires it; VBS's `Перем x;` doesn't have one.
  Inferred type carries through to `AddVariable` so `x.` chain
  autocomplete works.
* **Module body uses `allowSingleStmt = false`.** No `{` opener but
  not single-stmt. Only control-structure callsites use the default.
* **`StyleSetFont(STYLE_DEFAULT, ...)` BEFORE `StyleClearAll`.** The
  cascade propagates STYLE_DEFAULT to every style index so untouched
  styles (annotations, indent guides, BRACELIGHT/BRACEBAD) start
  with the user's font — proportional zoom requires same base size.

## Addendum — late-PM (2026-05-10, third pass)

Bug pass on the morning + PM IntelliSense work, plus the VBS → VES

### 1. IntelliSense — three follow-up fixes

**(a) Function body parsed as single-statement.** `CompileFunction`
called `CompileBlock(/*default*/)` after consuming the param list,
so a function body fell into single-statement mode and exited after
one `;`. Effect: any inline lambda inside a named function leaked
its `EndFunction` to the top-level parser → `CompileModule` failed
on every keystroke past the first such function. Fixed by passing
`/*allowSingleStmt=*/false` in `CompileFunction` and in the lambda
branch of `GetExpression`.

**(b) `m_declPos` plumbed through implicit-decl.** `var` is rare in
script code; the typical pattern is `a = expr;` and the LHS
auto-declares `a`. Without a declaration position the autocomplete
saw `a` everywhere — including lines above the implicit decl, which
was the user-visible "shows variable above its declaration" bug.

`ibPrecompileVariable::m_declPos` records the source-text offset of
the (auto-)declaring reference; the LoadAutoComplete walker filters
out variables whose `m_declPos > caretPos`. Parameters and context
injections keep `m_declPos = 0` so they're always visible.

Wired through:
* `ibPrecompileContext::AddVariable` and `GetVariable` accept
  `int declPos = 0`.
* `ibPrecompileCode::GetVariable(name, checkError, declPos)` wrapper.
* `GetCurrentIdentifier` SET branch passes `nStartPos` so the
  position points at the LHS occurrence (the de-facto declaration).

**(c) Empty dropdown at file top.** `PrepareExpression` greedily
captures the full identifier under the caret, so at column 0 of an
empty/early line `currentWord` came out non-empty. The list filter
then matched nothing and the dropdown showed empty. Fix:
`if (lenEntered == 0) currentWord = wxEmptyString;` in
`StartAutoComplete`. The user types forward to filter live; nothing
to match against at start-of-word means show all candidates.

### 2. Fold parser — unify CES + VES

The PM addendum's CES soft-statement work flipped the fold parser
into a brace-only mode for CES, dropping the keyword fold path.
Effect: switching the syntax-mode dropdown live (codeRunner) made
all `Procedure ... EndProcedure` / `If ... EndIf` folds vanish on
legacy keyword-style sources.

Rewrote `ibFoldLevelParser::CalcFoldLevel`: both modes share the
keyword-fold openers (Procedure / Function / If / While / For /
Try) and matching `End*` closers. CES adds `{ ... }` brace folds on
top — a hybrid `Procedure foo() { … } EndProcedure` folds at both
levels and the trailing-unclosed-fold trim in `UpdateFoldLevel`
prunes the keyword leg if the closer is missing in pure CES style.
