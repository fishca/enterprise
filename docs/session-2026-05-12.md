### #11 — Lazy block-syntax LINQ (deferred)

Block syntax `var r = from x in arr where p(x) select s(x);` currently emits inline OPER_FOREACH + `__r.Add(...)` into an Array — eager materialisation. For 10M-row queries this eats 10M ibValues in RAM (per LINQ pivot risk #2).

Options considered:
- **A. Compile-time rewrite to chain**: `from..where..select..` → `arr.Where(λ).Select(λ)`. Multi-from needs SelectMany; let needs transparent identifiers; join already has chain analog. ~500 LoC compile refactor.
- **B. Block-syntax state generator**: parser builds a `ibValueBlockQueryState` rather than emitting inline opcodes. Foreach body lives in synthesized lambda. Comparable invasiveness to A.
- **C. Opt-in `lazy` keyword**: `var q = from lazy x in arr ...`. ~200 LoC. Keeps eager as default.
- **D. Defer**: document the limitation; chain syntax already provides lazy alternative for users hitting RAM pressure; SQL pushdown for DB-backed sources is a separate (massive) arc.

**Chose D**: working copy is experimental, no concrete user pressure yet, chain syntax fills the gap. Revisit when:
- User explicitly hits OOM on a block-syntax query.
- SQL pushdown design starts (related concerns).
- LINQ surface stabilises (don't pile architectural changes on shifting foundation).

## Decisions to remember

* **Closure capture mechanism**: per-frame heap-promotion via `make_shared<ibRunContext>` + `enable_shared_from_this`. Lambda holds `vector<shared_ptr<ibRunContext>> m_capturedFrames`. OPER_CALL_VAL wires shim's m_pppArrayList[1..N] from captured frames per-invoke; OPER_CALL/OPER_CALL_L splits based on compile-time `m_needsHeapFrame` to keep regular calls probe-free.
* **Lexical vs dynamic parent**: `m_parentRunContext` for a lambda's frame points at its LEXICAL parent (first captured frame), not its dynamic caller. Set in OPER_CALL_VAL using `fn->m_capturedFrames[0].get()`. Enables triple-nested lambda capture walks via chain.
* **ibNumber out of union**: conditional-heap-owners can't live in `ibValue::union`. Test: type has non-trivial dtor / operator= / heap-pointer → vne union.
* **TYPE_FUNCTION/TYPE_ITERATOR specialisation**: replace RTTI `ConvertToType<T>()` (dynamic_cast walk via wxClassInfo) with typed-tag check + static_cast where T is a hot dispatch target. ibValueFunction/Iterator ctors stamp m_typeClass; AsFunction/AsIterator do tag check + walk one TYPE_REFFER hop. New TYPE_* tags need adding to all switch sites (Copy/Move/IsEmpty/Compare/GetTypeIDByRef skip-list).
* **Lambda invocation from host**: `InvokeLambdaWithArg(callable, arg, retVal)` is public API. Used by aggregations (`SumWithSelector` etc). Returns false if callable isn't a lambda — caller decides fallback.
* **Hash + insertion order pattern**: when grouping or hashing ibValues, use hybrid `std::map<ibValue, size_t, KeyCmp>` for O(log N) lookup + `std::vector<pair<key, vector<elem>>>` for insertion-order preservation. Pure std::map would re-sort by key.
* **Average rounds to 6 digits by default**. `ibNumber::operator/=` produces ~30 fractional digits via kExtra=30 in DivLongDivision; user-facing Average needs cosmetic truncation. Sum/Count manual division stays at full precision.
* **`InvokeLambdaWithArg` arg is taken by non-const ref**: lambda may have ByRef params and write back. Host caller should pass a local copy if it doesn't want mutation propagation.

## Addendum — evening: opcode + dispatch + editor polish

### Opcode renames + `OPER_CALL_LINQ` opcode

Trio rename for clarity:
- `OPER_CALL_L`   → `OPER_CALL_CLOSURE` (named call with heap-promoted frame for callees holding inner-lambda captures).
- `OPER_CALL_M`   → `OPER_CALL_METHOD`  (per-class method dispatch by name-string).
- `OPER_CALL_VAL` → `OPER_CALL_LAMBDA`  (dynamic call — target read from slot, must wrap ibValueFunction).

New `OPER_CALL_LINQ` opcode: chain-syntax LINQ method emit
(`arr.Where(λ)`) now reads the `ibLinqMethod` enum id straight from
`m_param3.m_numIndex` (no const-pool name, no `FindMethod` walk).
Compile-side detects LINQ method names via `ibValue::FindLinqMethodByName`
at chain-method emit site (compileCode.cpp). Runtime dispatcher
calls virtual `ibValue::DispatchLinqMethod` directly. The previous
`kLinqMethodMask` fallback inside `OPER_CALL_METHOD` was retired —
no legacy blobs reach runtime with a LINQ name there (AOT v10 forces
recompile).

AOT format version bumped 9 → 10: opcode insertion shifted every
subsequent opcode's numeric value, so v9 cache rows would dispatch
the wrong handler. v9 readers reject v10 blobs and vice versa →
safe recompile + repopulate.

### CES fold-keyword bug fix

User-reported (2026-05-12 evening): in CES, Allman-style
`Function\n{\n body\n}` body landed at indent 2 instead of 1.

Cause: `KEY_PROCEDURE` / `KEY_FUNCTION` / `KEY_IF` / etc. opened
keyword folds, but the `End*` closers are filtered out by
`IsAllowedKey` in CES (only `}` is a legal closer). Orphan opens
stacked +1 on every body line — combined with brace folds, body
indent doubled.

Fix: in CES, suppress keyword fold opens for control-flow
keywords. Only braces drive fold structure; keywords are pure
syntactic markers. LINQ keywords (KEY_FROM / KEY_SELECT /
KEY_GROUP) kept — their closers stay keywords in CES.
